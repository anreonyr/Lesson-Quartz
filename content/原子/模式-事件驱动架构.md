# 事件驱动架构

基于事件的异步通信模式，组件之间通过事件进行解耦。

## 核心概念

- **事件**：发生的事情（如用户注册、订单创建）
- **发布者**：产生事件的组件
- **监听者**：响应事件的组件
- **事件总线/消息队列**：事件的传递中介

## 架构图

```
┌──────────────┐      事件      ┌──────────────┐
│   事件发布者 │  ──────────→   │   事件监听者 │
│  (Publisher) │                │  (Subscriber)│
└──────────────┘                └──────────────┘
```

## Python示例

```python
def on_user_registered(user):
    send_welcome_email(user)
    add_to_newsletter(user)
    log_registration(user)

class EventBus:
    def __init__(self):
        self.listeners = {}

    def subscribe(self, event_type, callback):
        self.listeners.setdefault(event_type, []).append(callback)

    def publish(self, event_type, data):
        for callback in self.listeners.get(event_type, []):
            callback(data)

event_bus = EventBus()
event_bus.subscribe('user_registered', on_user_registered)
event_bus.publish('user_registered', {'name': '张三'})
```

## 优点

- 高度解耦：发布者和监听者互不感知
- 易于扩展：可以添加新的监听者而不修改发布者
- 响应迅速：异步处理，无需等待

## 缺点

- 调试困难：事件流不直观
- 复杂度高：需要消息队列等基础设施
- 顺序难保证：事件处理顺序不确定

## 适用场景

- 实时系统（如聊天、通知）
- 需要解耦的复杂业务
- 大数据处理管道
