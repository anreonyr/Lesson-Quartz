# 分层架构

最经典的架构模式，将应用分成多个层次，每层只关注自己的职责。

## 层次结构

```
┌─────────────────────────────────┐
│       表示层（Presentation）    │  ← 处理用户界面和请求
├─────────────────────────────────┤
│       业务层（Business）        │  ← 处理业务逻辑
├─────────────────────────────────┤
│       数据访问层（Data Access） │  ← 处理数据库操作
├─────────────────────────────────┤
│         数据层（Data）          │  ← 数据库、文件系统
└─────────────────────────────────┘
```

## 各层职责

- **表示层**：接收用户请求，返回响应（Controller/View）
- **业务层**：处理核心业务逻辑（Service）
- **数据访问层**：与数据库交互（Repository/DAO）
- **数据层**：存储数据（Database）

## Python示例（Django）

```python
# 表示层：urls.py + views.py
def user_list(request):
    users = get_users()  # 调用业务层
    return render(request, 'users.html', {'users': users})

# 业务层：services.py
def get_users():
    return User.objects.all()  # 调用数据访问层

# 数据访问层：models.py
class User(models.Model):
    name = models.CharField(max_length=100)
```

## 优点

- 结构清晰，易于理解和维护
- 每层职责明确
- 便于团队协作

## 缺点

- 层次过多会影响性能
- 容易导致代码冗余

## 适用场景

- 企业级Web应用
- 需要清晰结构的项目
